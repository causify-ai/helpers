    Writing Docs#

    Summary#

    • This document describes how to write markdown for:

      • Internal documentation
      • Blog entries
      • Tutorials

    • This document is geared towards humans, while the style guide [//helpers/docs/code_guidelines/all.coding_style_guidelines.reference.md] is for non-human users (e.g., linter and ai_review.py)

    Meta#

    • Each of the suggestions below should be a level 3 heading so that it's easy to point to it with a link

    Conventions#

    Layout Rules#

    • Visual Structure

      • Use clear headings
      • Use nested bullets for hierarchy
        • 1 idea per bullet
      • Separate sections logically

    • Clarity

      • Use simple language
      • Define terms immediately
      • Avoid ambiguous phrasing

    • Headings

      • Use # for main topics
      • Use ## for subtopics
      • Keep headings short and descriptive

    • Bullets

      • - for main bullets
      • Indent for sub-bullets
      • Each bullet = 1 idea

    • Diagrams

      • Use diagrams instead of wordy explanations
      • Use mermaid, graphviz, or tikz fenced code blocks
      • Prefer flowcharts, sequence diagrams, or graphs

    Make no assumptions on the user's knowledge#

    • Nothing is obvious to somebody who doesn't know

    Hold user's hand#

    • Add ways to verify if a described process worked
      • E.g., "do this and that, if this and that is correct should see this"
    • Have a trouble-shooting procedure
      • One approach is to always start from scratch

    Always use Linter#

    • Most cosmetic suggestions are handled by our Linter

      • Run it after changes
      • Use i lint --files="your_file_name"

    • Avoid mixing manual edits and Linter runs

      • Run Linter and commit changes separately

    • Add a table of contents

      • Markdown doesn't auto-generate TOC
      • Run Linter to build TOC and place it at the top

    • If Linter errors occur, file an issue with examples

    Format of each doc#

    • There should be only one level 1 header with the title of the document
      • The title is used by mkdocs

    • There should be a summary with a short summary in bullets of the document

    • Good: ```text

      <a class="headerlink" href="#_1" title="Permanent link">#</a></h2> <h3 id="summary_1">Summary<a class="headerlink" href="#summary_1" title="Permanent link">#</a></h3> <ul> <li>This document contains ...</li> </ul> <h3 id="resources">Resources<a class="headerlink" href="#resources" title="Permanent link">#</a></h3> <h3 id="last_review">Last review<a class="headerlink" href="#last_review" title="Permanent link">#</a></h3> <ul> <li>GP on 2024-04-20</li> <li>Paul on 2024-03-10 ```</li> </ul> </li> </ul> <h4 id="use_80_columns_formatting_for_markdown_files">Use 80 columns formatting for markdown files<a class="headerlink" href="#use_80_columns_formatting_for_markdown_files" title="Permanent link">#</a></h4> <ul> <li>The <code>linter</code> takes care of reflowing the text</li> <li><code>vim</code> has a <code>:gq</code> command to reflow the comments</li> <li>There are plugins for PyCharm and VisualStudio</li> </ul> <h4 id="use_good_vs_bad">Use good vs bad<a class="headerlink" href="#use_good_vs_bad" title="Permanent link">#</a></h4> <ul> <li> <p>Make examples of "good" ways of doing something and contrast them with "bad" ways using the following format, e.g.,</p> <p><code>` - Good:</code>markdown ... ```</p> <ul> <li>Bad: <code>markdown ...</code> ````</li> </ul> </li> </ul> <h4 id="use_an_empty_line_after_heading">Use an empty line after heading<a class="headerlink" href="#use_an_empty_line_after_heading" title="Permanent link">#</a></h4> <ul> <li> <p>Leave an empty line after a heading to make it more visible, e.g.,</p> <ul> <li>Good ```markdown</li> </ul> <h2 id="very_important_title">Very important title<a class="headerlink" href="#very_important_title" title="Permanent link">#</a></h2> <ul> <li> <p>Less important text ```</p> </li> <li> <p>Bad ```markdown</p> </li> </ul> <h2 id="coming_through_ive_big_important_things_to_do">Coming through! I've big important things to do!<a class="headerlink" href="#coming_through_ive_big_important_things_to_do" title="Permanent link">#</a></h2> <ul> <li>... and his big important wheels got STUCK! ```</li> </ul> </li> <li> <p>Our Linter automatically takes care of this</p> </li> </ul> <h4 id="bullet_lists">Bullet lists<a class="headerlink" href="#bullet_lists" title="Permanent link">#</a></h4> <ul> <li>Use bullet lists since they represent the thought process, force people to focus on short sentences (instead of rambling wall-of-text), and relation between sentences</li> <li>E.g., ```markdown<ul> <li>This is thought #1<ul> <li>This is related to thought #1</li> </ul> </li> <li>This is thought #2<ul> <li>Well, that was cool!</li> <li>But this is even better ```</li> </ul> </li> </ul> </li> <li>Use <code>-</code> instead of <code>*</code> or circles</li> <li>Linter automatically enforces this</li> </ul> <h4 id="use_the_right_syntax_highlighting">Use the right syntax highlighting<a class="headerlink" href="#use_the_right_syntax_highlighting" title="Permanent link">#</a></h4> <ul> <li>When using a block of code use the write syntax highlighting<ul> <li>Code (```python)</li> <li>Dirs (e.g.,<code>/home/users</code>)</li> <li>Command lines (e.g., <code>> git push</code> or <code>docker> pytest</code>)</li> <li>Bash <code>bash > git push</code></li> <li>Python <code>python if __name__ == "__main__": predict_the_future() print("done!")</code></li> <li>Markdown <code>markdown ...</code></li> <li>Nothing <code>verbatim ....</code></li> </ul> </li> </ul> <h4 id="indent_code_style">Indent <code>code</code> style<a class="headerlink" href="#indent_code_style" title="Permanent link">#</a></h4> <ul> <li>GitHub / Pandoc seems to render incorrectly a code block unless it's indented over the previous line</li> </ul> <h4 id="embed_screenshots_only_when_strictly_necessary">Embed screenshots only when strictly necessary<a class="headerlink" href="#embed_screenshots_only_when_strictly_necessary" title="Permanent link">#</a></h4> <ul> <li>Avoid to use screenshots whenever possible and use copy-paste of text with the right highlighting</li> <li>Sometimes you need to use screenshots (e.g., plots, website interface)</li> </ul> <h4 id="improve_your_written_english">Improve your written English<a class="headerlink" href="#improve_your_written_english" title="Permanent link">#</a></h4> <ul> <li>Use English spell-checker</li> <li>Type somewhere where you can use several choices:<ul> <li><a href="https://www.grammarly.com/">Grammarly</a></li> <li>ChatGPT</li> <li><a href="https://www.languagetool.org/">LanguageTool</a></li> </ul> </li> <li>This is super-useful to improve your English since you see the error and the correction<ul> <li>Otherwise you will keep making the same mistakes forever</li> </ul> </li> </ul> <h4 id="make_sure_your_markdown_looks_good">Make sure your markdown looks good<a class="headerlink" href="#make_sure_your_markdown_looks_good" title="Permanent link">#</a></h4> <ul> <li> <p>Compare your markdown with others already published</p> </li> <li> <p>You can:</p> <ul> <li>Check in the code in a branch and use GitHub to render it</li> <li>Use IDEs to edit, which also renders it side-by-side</li> </ul> </li> </ul> <h4 id="capitalize_headings">Capitalize headings<a class="headerlink" href="#capitalize_headings" title="Permanent link">#</a></h4> <ul> <li>Headings titles should be like <code>Data Schema</code> not <code>Data schema</code><ul> <li>This is automatically enforced by the <code>linter</code></li> </ul> </li> </ul> <h4 id="update_the_last_review_tag">Update the <code>Last review</code> tag<a class="headerlink" href="#update_the_last_review_tag" title="Permanent link">#</a></h4> <ul> <li>When you read/refresh a file update the last line of the text <code>verbatim ## Last review - GP on 2024-04-20 - Paul on 2024-03-10</code></li> </ul> <h4 id="comment_the_code_structure">Comment the code structure<a class="headerlink" href="#comment_the_code_structure" title="Permanent link">#</a></h4> <ul> <li>When you <code>docs/documentation_meta/all.diataxis.explanation.md</code></li> </ul> </li> <li>Where<ul> <li><code>component</code> is one of the software components (e.g., <code>datapull</code>, <code>dataflow</code>)</li> <li><code>audience</code> is the target audience (e.g., <code>all</code>, <code>ck</code>)</li> <li><code>topic</code> is the topic of the file</li> <li>The topic of a "how to guide" should have a verb-object format<ul> <li>E.g., <code>docs/oms/broker/ck.generate_broker_test_data.how_to_guide.md</code></li> </ul> </li> <li>The topic of a "reference" is often just a <li>Active voice is shorter than passive voice</li> <li> <p>Readers convert passive voice to active voice</p> </li> <li> <p>Good: "You can change these configuration by ..."</p> </li> <li>Bad: "These configurations can be changed by ..."</li> </ul> <h4 id="format_for_easy_reading">Format for easy reading<a class="headerlink" href="#format_for_easy_reading" title="Permanent link">#</a></h4> <ul>
