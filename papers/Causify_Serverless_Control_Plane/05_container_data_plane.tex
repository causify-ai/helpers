  \section{Container Data Plane Integration}
  \label{sec:data_plane_integration} If the control plane is the cortex,
  the data plane is the muscle: it runs services, workflows, and compute-heavy
  tasks. In \system{}, the data plane is intentionally heterogeneous,
  because the workloads are heterogeneous.

% ==============================================================================
  \subsection{Kubernetes services on EKS: long-running platform components}
  Long-running components live on Kubernetes (EKS) \cite{eks_docs}. This
  includes web-facing services and platform internals.

  A representative example is the Causify Console deployment: it runs
  behind an internal AWS ALB ingress (TLS enforced, WAF attached, Shield
  enabled, and access logs streamed to S3). The same patterns appear for
  Airflow's webserver ingress, which is also internal and protected.

  \paragraph{Airflow on Kubernetes (workflow plane).}
  Airflow is deployed in the \texttt{airflow} namespace with separate components
  (e.g., scheduler, webserver, dag-processor) and a \texttt{git-sync}
  deployment to pull DAGs. The scheduler is autoscaled via HPA (CPU and
  memory utilization targets), and operational hygiene is handled via
  cleanup cronjobs.

  Secrets are integrated using an ESO service account annotated for IRSA
  (\texttt{eks.amazonaws.com/role-arn}) to fetch secrets from AWS Secrets
  Manager. This allows Kubernetes workloads to consume secrets without
  static credentials.

  In short: Kubernetes runs the things that must keep running.

% ==============================================================================
  \subsection{Module backends as Lambda services: Horizon and Grid}
  Some module backends are implemented as dedicated Lambda functions (Horizon/Grid)
  that implement their own routing tables and connect directly to a Postgres
  database (using Secrets Manager and a proxy endpoint via environment
  variables).

  These backends are deliberately strict about tenant scoping. For
  example, the Horizon Lambda extracts \texttt{schema\_prefix} from
  query parameters and refuses to operate without it (treating it as required
  input). It then derives schema names such as \texttt{\{schema\_prefix\}\_core}
  internally.

  This makes an architectural point:
  \begin{quote}
    \textbf{The control plane decides scope; the module backend enforces
    it locally.}
  \end{quote}

  The control plane ensures \texttt{schema\_prefix} is derived from authenticated
  context and injected downstream; the module backend ensures it does
  not run without scope.

% ==============================================================================
  \subsection{Airflow $\rightarrow$ ECS/Fargate: burst compute without cluster
  debt}
  Not all compute belongs in Kubernetes. For bounded, bursty jobs (pipelines,
  model runs, data downloads), Airflow dispatches ECS tasks (Fargate by
  default) \cite{ecs_runtask,fargate_docs}.

  Airflow DAGs for Horizon and Grid use a helper that produces a configured
  ECS operator. The helper sets CPU/memory, ephemeral storage, and VPC
  networking (private subnets and security groups by region). It also wraps
  the operator with retry logic for known ECS capacity failures.

  \begin{samepage}
    \begin{lstlisting}[style=python,caption={Airflow launches ECS tasks via a custom operator that retries known capacity/provisioning failures.},label={lst:ecs_retry}]
class EcsRunTaskOperator_withRetry(EcsRunTaskOperator):
    _KNOWN_ERRORS = ["Capacity is unavailable at this time."]
    _MAX_NUM_ATTEMPTS = 2
    _RETRY_DELAY_SECS = 15
\end{lstlisting}
  \end{samepage}

  \begin{samepage}
    \begin{lstlisting}[style=python,caption={The ECS task operator is configured with VPC networking, resources, and long waiter timeouts (pragmatic fix for real-world behavior).},label={lst:ecs_operator_cfg}]
task = EcsRunTaskOperator_withRetry(
  task_definition=task_definition,
  launch_type="FARGATE",
  overrides={
    "cpu": str(cpu_units),
    "memory": str(memory_mb),
    "ephemeralStorage": {"sizeInGiB": ephemeralStorageGb},
    "containerOverrides": [{"name": task_definition, "command": task_cmd}],
  },
  network_configuration={"awsvpcConfiguration": {...}},
)
\end{lstlisting}
  \end{samepage}

  This pattern yields a clean separation: Airflow orchestrates workflows;
  ECS/Fargate executes job containers; Kubernetes remains focused on
  long-lived services.

  \subsection{Why this mixed data plane is intentional}
  A single execution substrate is aesthetically pleasing and
  operationally tempting. It is also how you end up running batch workloads
  in the same cluster as your interactive services and wondering why everything
  is on fire.

  The platform embraces the reality that:
  \begin{itemize}
    \item Kubernetes is excellent for long-running services and controllable
      scaling.

    \item Fargate is excellent for isolated burst compute and operational
      simplicity.

    \item Lambda is excellent for narrowly scoped APIs and fast fan-out.
  \end{itemize}

  The art is not choosing one. The art is choosing boundaries.

  \subsection{Domain Module Lambdas: Horizon and Grid as Tenant-Scoped Query
  Planes}
  The platform uses dedicated module backends for Horizon and Grid as
  AWS Lambda services. This is not ``Lambda because serverless is trendy.''
  It is Lambda because these modules behave like \emph{query planes}:
  they must respond quickly, enforce scoping strictly, and scale to
  spiky interactive traffic without dragging an entire Kubernetes
  service along for every request.

  \paragraph{Horizon Lambda.}
  Horizon is opinionated and explicit: every request must name its
  tenant scope via \texttt{schema\_prefix}. If it is missing, the request
  is rejected early. The Lambda then derives its working schema (\texttt{core\_schema
  = \{schema\_prefix\}\_core}), and constructs a service layer that
  executes queries against tenant-scoped tables. To keep the system calm
  under transient failures, Horizon uses short retry backoff and
  structured JSON logs to keep debugging deterministic instead of
  mystical.

  \begin{samepage}
    \begin{lstlisting}[style=python,caption={Horizon Lambda enforces tenant scope explicitly: \texttt{schema\_prefix} is required and used to derive \texttt{core\_schema}.},label={lst:horizon_scope}]
q = self.event.get("queryStringParameters") or {}
self.schema_prefix = q.get("schema_prefix")
if not self.schema_prefix:
    raise BadRequestError("schema_prefix is required")
self.core_schema = f"{self.schema_prefix}_core"
\end{lstlisting}
  \end{samepage}

  \paragraph{Database access: Secrets Manager + RDS Proxy.}
  Operationally, each module Lambda follows a three-step pattern to reach
  tenant data: (i) fetch database credentials from AWS Secrets Manager
  \cite{aws_secrets_manager} at cold start, (ii) connect through an RDS Proxy
  endpoint over TLS, and (iii) execute parameterized SQL using a small, testable
  service layer.

  \begin{samepage}
    \begin{lstlisting}[style=python,caption={Database connection via Secrets Manager and RDS Proxy: credentials are never embedded, and the proxy manages connection pooling across Lambda invocations.},label={lst:rds_proxy}]
_SECRET_ARN = os.environ["DB_SECRET_ARN"]
_PROXY      = os.environ["PROXY_ENDPOINT"]
_DB_NAME    = os.environ["DB_NAME"]

@retryable
def build() -> PgClient:
    sec = boto3.client("secretsmanager")
          .get_secret_value(SecretId=_SECRET_ARN)["SecretString"]
    creds = json.loads(sec)
    conn = psycopg2.connect(
        host=_PROXY, dbname=_DB_NAME,
        user=creds["username"], password=creds["password"],
        sslmode="require",
    )
    return PgClient(conn)
\end{lstlisting}
  \end{samepage}

  This completes the scope propagation story end-to-end: identity is validated
  in the control plane, a reusable \scopeenv{} is attached to downstream
  execution (i.e., \texttt{schema\_prefix} is derived and injected into the
  delegation event), the module Lambda uses that prefix to derive \texttt{\{schema\_prefix\}\_core},
  and the database connection itself goes through a managed proxy that handles
  connection pooling and IAM-based access. At no point does a client
  parameter touch the database path.

  \paragraph{Grid Lambda.}
  Grid is similarly scope-driven, but its shape is slightly different: it
  serves time-series analytics, predictions, and derived signals, and
  uses the schema prefix to select the correct core tables. The design is
  intentionally boring: a mapping of features to tables and columns is
  built around the tenant's core schema, which avoids hardcoding tenant-specific
  logic into application code.

  \begin{samepage}
    \begin{lstlisting}[style=python,caption={Grid Lambda uses \texttt{schema\_prefix} to select tenant-scoped core tables.},label={lst:grid_scope}]
core_table = f"{schema_prefix}_core.load_predictions"
prob_table = f"{schema_prefix}_core.threshold_cross_probability"
\end{lstlisting}
  \end{samepage}

  \paragraph{Why Lambdas for these modules?}
  Because these modules sit at a boundary where traffic is bursty and latency
  matters: interactive dashboards behave like a strobe light, not a steady
  lamp. Lambda matches that rhythm. Kubernetes excels at steady-state services
  and background work; module Lambdas excel at sharp edges where cold
  requests show up at inconvenient times.

  \paragraph{The contract with the control plane.}
  Neither Horizon nor Grid is allowed to ``discover'' its tenant. Both
  consume scope. In practice, \texttt{schema\_prefix} is injected by the
  control plane and forwarded through delegating routers. This is the
  anti-confused-deputy rule in executable form: \emph{tenancy is derived
  from identity, not from client parameters.}

  \subsection{DataMap Studio: Subject $\rightarrow$ Analysis $\rightarrow$
  Graph, With Memory}
  DataMap is the platform's causal exploration workbench: a guided flow that
  turns datasets into structured hypotheses and causality graphs without
  forcing users to juggle a zoo of unrelated artifacts.

  The studio is organized around a simple hierarchy:

  \begin{center}
    \textbf{Subject} $\rightarrow$ \textbf{Analysis} $\rightarrow$ \textbf{Graph}
    \quad (and each Graph can have \textbf{Versions})
  \end{center}

  This hierarchy is not aesthetic; it is a data model that prevents sprawl.
  It supports: (i) multiple analyses per subject, (ii) multiple graph
  versions per analysis, and (iii) per-user, multi-session workflows
  without multiplying tables per feature.

  \paragraph{Refine-in-place, but keep history.}
  DataMap behaves like a whiteboard that remembers. Users can refine
  nodes/edges in place, then view version history and roll back if an iteration
  gets weird. This makes experimentation safe: curiosity is encouraged,
  and regret is supported.

  \paragraph{Any data source.}
  DataMap treats ingestion as a pluggable boundary. A dataset upload is just
  one implementation of ``data source.'' The same workflow can support
  connectors and query-backed sources without changing the studio's
  conceptual model.

  \paragraph{Multi-session without table sprawl.}
  Instead of creating new tables for every ``session'' or feature variant,
  DataMap stores sessions and versions as \emph{keyed artifacts} attached
  to the Subject/Analysis hierarchy. In practice, this supports:
  \begin{itemize}
    \item multiple concurrent sessions per user,

    \item multiple users working in parallel,

    \item and a clean audit trail of graph evolution.
  \end{itemize}

  The result is a studio that feels fluid while staying structurally disciplined.
  It is creativity with guardrails.

  \paragraph{One table, several item types.}
  DataMap uses a single DynamoDB table per environment (e.g., \texttt{causify\_console\_datamap\_\{env\}})
  that stores a small set of item types: \texttt{SUBJECT}, \texttt{ANALYSIS},
  \texttt{GRAPH} (versioned), \texttt{SESSION}, and \texttt{TURN}. The
  partition/sort keys are compositional, so the hierarchy is queryable without
  joins.

  \begin{samepage}
    \begin{lstlisting}[style=dynamo,caption={DataMap key patterns (verbatim structure).},label={lst:datamap_keys_v2}]
  SUBJECT
    PK: TENANT#<tenant_id>#SUBJECT#<subject_id>
    SK: SUBJECT
    Attrs: subject_type, subject_key (JSON), display_name, description,
           s3_uri_metadata, status, created_ts, created_by, ...

  ANALYSIS
    PK: TENANT#<tenant_id>#SUBJECT#<subject_id>
    SK: ANALYSIS#<analysis_id>    (analysis_id = ULID/UUID)
    Attrs: title, created_ts, created_by_user, last_graph_id, tags[]

  GRAPH (versioned output)
    PK: TENANT#<tenant_id>#SUBJECT#<subject_id>#ANALYSIS#<analysis_id>
    SK: GRAPH#<version_ts_desc>   (store as ~ts or negative epoch for newest-first)
    Attrs: graph_id (ULID), graph_json OR s3_key_graph_json,
           summary[], changes[] (refine deltas),
           model, prompt_version, temperature, trace_id,
           created_by_user, session_id, size_bytes, optional ttl

  SESSION (per-user working session)
    PK: TENANT#<tenant_id>#USER#<user_id>
    SK: SESSION#<session_id>
    Attrs: subject_id, analysis_id (optional), started_ts, last_graph_id,
           status (active|closed), ttl

  TURN (multi-turn chat/history)
    PK: TENANT#<tenant_id>#SESSION#<session_id>
    SK: TURN#<seq>   (001, 002, ... or timestamp)
    Attrs: role (user|system|assistant|...), content, result_graph_id (optional),
           trace_id, optional ttl
  \end{lstlisting}
  \end{samepage}

  The key design is compositional: partition keys encode tenancy and hierarchy;
  sort keys encode artifact identity and time.

  \paragraph{Versioning without drama.}
  Graph versions store either embedded graph JSON or a pointer to an S3 key.
  They also store refine deltas (\texttt{changes[]}), summaries, and trace
  metadata (\texttt{trace\_id}, \texttt{prompt\_version}, \texttt{temperature})
  so the system can explain not just \emph{what} changed, but \emph{why}.

  \paragraph{Indexes that match the product.}
  The GSIs follow the user journeys:
  \begin{itemize}
    \item \textbf{Recent sessions per user:} \texttt{GSI1PK = TENANT\#<tenant\_id>\#USER\#<user\_id>},
      \texttt{GSI1SK = TS\#<started\_ts>}.

    \item \textbf{Latest graph per analysis:} \texttt{GSI2PK = TENANT\#<tenant\_id>\#SUBJECT\#<subject\_id>\#ANALYSIS\#<analysis\_id>},
      \texttt{GSI2SK = GRAPH\#<version\_ts\_desc>} (query with \texttt{limit=1}).

    \item \textbf{All graphs by user:} \texttt{GSI3PK = USER\#<user\_id>},
      \texttt{GSI3SK = TS\#<version\_ts\_desc>}.
  \end{itemize}

  \paragraph{Storage symmetry with S3.}
  Datasets are stored read-only in S3; exports are organized by tenant/subject/analysis/graph
  prefixes, with server-side encryption and blocked public access.
  DynamoDB stores the "shape" and history; S3 stores the heavy artifacts.
  TTL can be used for demo cleanup where appropriate.

  This model yields to our desired studio behavior: refine in place,
  view history, restore earlier versions, support multiple analyses per
  subject, and allow many concurrent user sessions---all without creating
  new tables every time the product team has an idea.

  The studio remains fast because the ``truth'' is indexable, and the ``bulk''
  is object storage.

  \subsection{Tenant meta as data: membership, connectors, billing,
  preferences, audit}
  Beyond modules and compute, a multi-tenant platform needs "tenant meta"
  primitives: who belongs to a tenant, which connectors are configured,
  whether billing allows a module, and what defaults a user prefers. We
  model this in DynamoDB with small, purpose-built tables and GSIs that match
  access patterns:

  \begin{itemize}
    \item \textbf{Membership/seats:} \texttt{PK=tenant\_id}, \texttt{SK=user\_id}
      with GSI by email to resolve invites and multi-tenant users.

    \item \textbf{Invites:} unguessable \texttt{invite\_token} as primary
      key; on acceptance, create membership row and add Cognito group \texttt{\{tenant\_id\}:\{role\}}.

    \item \textbf{Connector state:} \texttt{PK=tenant\_id}, \texttt{SK=connector\_key}
      storing status, last sync timestamps, and a reference to credentials
      stored in Secrets Manager (not raw secrets).

    \item \textbf{Billing:} one row per tenant with GSIs for Payment
      Gateway (e.g., Stripe) customer/subscription lookup so webhooks
      can map events $\rightarrow$ tenant deterministically.

    \item \textbf{Preferences:} \texttt{PK=tenant\_id}, \texttt{SK=user\_id}
      for default modules, locale/theme, etc.

    \item \textbf{Audit log:} \texttt{PK=tenant\_id}, \texttt{SK=timestamp\#uuid}
      with action\_type and payload.
  \end{itemize}

  The platform benefit is operational: these models make onboarding,
  licensing, connector operations, and UX defaults \emph{data operations}
  rather than application forks.

  A reusable pattern emerges in connector-heavy multi-tenant products (e.g.,
  CPG connectors such as Shopify/Amazon): \emph{keep the connector hot
  path API-driven and multi-tenant by design}, and use workflow engines only
  for offline analytics. Our ingestion architecture therefore mirrors
  the paper's thesis: \emph{thin control plane, heavy execution plane}.

    and enqueues job intents into SQS\@. The data plane executes on
    Kubernetes via three worker types: short-lived Jobs for Shopify
    backfills, Jobs/Deployments for Amazon syncs, and an always-running
    Deployment for Shopify real-time webhook consumption. All workers
    write raw data to S3 and normalized data to Postgres
    (\texttt{cpg\_stage}\,/\,\texttt{cpg\_core}). EventBridge provides
    cron scheduling for Amazon pulls and webhook routing for Shopify
    events.}
  \label{fig:connector_arch}

  \paragraph{Control plane responsibilities.}
  When a tenant connects a source (e.g., Shopify), the request flows
  through API Gateway into the platform control plane and a domain
  controller service. That service:
  \begin{itemize}
    \item completes OAuth/auth, validates scopes, and handles webhook registration
      via EventBridge,

    \item stores connector metadata in DynamoDB (tenant, external identifiers,
      status, last sync timestamps),

    \item stores access tokens in Secrets Manager,

    \item and enqueues job intents into SQS (e.g., \texttt{job\_type=historical\_backfill}
      or \texttt{incremental\_sync}) with \texttt{tenant\_id} and
      connector identifiers \cite{aws_sqs_docs}.
  \end{itemize}
  The UI hits \texttt{/v1/cpg/connectors/...} endpoints which read/write
  job state from Dynamo so the tenant can see \emph{sync in progress /
  completed / failed} per connector.

% ------------------------------------------------------------------------------
  \subsubsection{Data plane: historical backfill}
  For historical download, the data plane uses Shopify Bulk Operations to
  fetch large datasets efficiently. A \texttt{StartHistoricalPull} Lambda
  validates scopes, starts bulk operations, and stores \texttt{bulk\_operation\_id}
  and configuration in Dynamo. When Shopify completes the bulk operation,
  a webhook routes via EventBridge into a \texttt{ProcessBulkResult} Lambda
  \cite{aws_eventbridge_docs} that:
  \begin{itemize}
    \item downloads the JSONL result,

    \item writes it to \texttt{S3/raw/} (partitioned by shop+date),

    \item and writes staging extracts into \texttt{cpg\_stage}
      \cite{aws_s3_docs}.
  \end{itemize}
  Normalization into the canonical model is executed by either Step
  Functions, ECS ETL, or a container worker on Kubernetes (depending on workload
  shape and cost profile)
  \cite{aws_stepfunctions_docs,ecs_runtask,fargate_docs,eks_docs}.

% ------------------------------------------------------------------------------
  \subsubsection{Data plane: near-real-time (webhooks + always-running
  workers)}
  For ongoing updates, we combine Shopify webhooks with always-running workers
  on Kubernetes:
  \begin{itemize}
    \item Shopify calls our webhook URLs (API Gateway $\rightarrow$
      Lambda),

    \item the webhook handler validates the HMAC signature, maps shop
      domain $\rightarrow$ tenant/connector in Dynamo,

    \item and writes a compact event into SQS (optionally storing full
      payload in S3 for replay).
  \end{itemize}
  On Kubernetes, an always-running \texttt{shopify-realtime-worker}
  consumes events, hydrates details when needed, and applies upsert
  semantics into \texttt{cpg\_stage} / \texttt{cpg\_core}.

  To cover missed webhooks, we can run periodic reconciliation (EventBridge
  cron or Kubernetes cron): scan active connectors, enqueue an \texttt{incremental\_sync}
  job with a window since \texttt{last\_incremental\_sync\_at}, and
  process it with the same worker image. This yields near-real-time freshness
  without depending on Airflow or long-lived Lambdas.

% ------------------------------------------------------------------------------
  \subsubsection{Amazon follows the same shape}
  Amazon SP-API does not provide the same webhook story, so everything
  is scheduled: EventBridge triggers a small coordinator Lambda that lists
  active Amazon connectors and enqueues backfill/incremental jobs in SQS
  per seller account. A Kubernetes worker image handles both backfills (short-lived
  Jobs) and incremental syncs (Jobs or a Deployment), writing raw to S3 and
  populating \texttt{cpg\_stage} / \texttt{cpg\_core}.

% ------------------------------------------------------------------------------
  \subsubsection{Role of Airflow}
  Airflow is intentionally \emph{not} in the Shopify/Amazon datapull
  path. We use Airflow later for offline analytics and forecasting \emph{on
  top of} \texttt{cpg\_core} (nightly demand forecasts, promo elasticity
  jobs, heavy batch analysis), keeping ingestion API-driven and multi-tenant
  by design.

% ------------------------------------------------------------------------------
  \subsubsection{Tenant isolation in Postgres: tenant\_id + optional RLS
  safety net}
  Unlike schema-per-tenant systems, the CPG module uses shared schemas (\texttt{cpg\_stage},
  \texttt{cpg\_core}) with \texttt{tenant\_id BIGINT NOT NULL} on tenant-owned
  tables. The schema DDL makes this explicit across dimensions and facts
  (e.g., \texttt{cpg\_core.fact\_order\_line} includes \texttt{tenant\_id}):

  \newpage
  \begin{samepage}
    \begin{lstlisting}[style=sql,caption={Shared schemas + tenant\_id column.},label={lst:cpg_tenant_id}]
CREATE SCHEMA IF NOT EXISTS {schema_prefix}_stage;
CREATE SCHEMA IF NOT EXISTS {schema_prefix}_core;

CREATE TABLE IF NOT EXISTS {schema_prefix}_core.fact_order_line (
    order_line_pk BIGSERIAL PRIMARY KEY,
    tenant_id     BIGINT    NOT NULL,
    order_pk                BIGINT        NOT NULL REFERENCES {schema_prefix}_core.fact_order_header(order_pk),
    sku_id                  BIGINT        NOT NULL REFERENCES {schema_prefix}_core.dim_sku(sku_id),
    external_order_item_id  VARCHAR(255),        -- Shopify line_item.id, Amazon OrderItemId, etc.
    ...

    UNIQUE (tenant_id, order_pk, sku_id, COALESCE(external_order_item_id, ''))
);

CREATE INDEX IF NOT EXISTS fact_order_line_tenant_sku_idx
    ON {schema_prefix}_core.fact_order_line (tenant_id, sku_id);
\end{lstlisting}
  \end{samepage}

  At runtime, the safest pattern is belt-and-suspenders: queries explicitly
  filter by \texttt{tenant\_id}, and row-level security (RLS) can be enabled
  as a backstop. A common approach is to bind tenant scope into the DB
  session using a custom setting (GUC):

  \begin{samepage}
    \begin{lstlisting}[style=sql,caption={Optional RLS backstop: bind tenant scope into the session and enforce via policy.},label={lst:rls_policy}]
-- application sets scope per request
SET LOCAL app.tenant_id = '<tenant_id>';

ALTER TABLE {schema_prefix}_core.fact_order_line ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_fact_order_line
ON {schema_prefix}_core.fact_order_line
USING (tenant_id = current_setting('app.tenant_id')::BIGINT);
\end{lstlisting}
  \end{samepage}

  This is the same paper thesis in SQL form: execution may be fast, but scope
  must be explicit.

  \subsection{Optima: heavy compute as container services, governed by
  serverless wrappers}
  Some analytics do not fit inside Lambda without turning the deployment
  into a gym membership: NumPy/pandas/CVXPY-class workloads either
  exceed practical packaging limits or produce unacceptable cold starts.
  We therefore implement Optima-style modules as \emph{container-native
  compute} governed by the same control plane contracts as the rest of
  the platform.

  \paragraph{The pattern.}
  Optima follows a three-part architecture that is intentionally reusable:
  \begin{itemize}
    \item \textbf{Thin serverless wrapper:} validates identity, derives effective
      context, enforces licensing/seat gates, and emits a job intent (scope
      + parameters).

    \item \textbf{Heavy container compute:} executes optimization/simulation/forecast
      workloads on EKS/ECS/Fargate with warm dependencies and
      predictable runtime behavior.

    \item \textbf{Artifact-first I/O:} large inputs/outputs move via S3
      (often with presigned upload URLs), keeping API Gateway and the control
      plane slim and reliable \cite{aws_s3_presigned,aws_s3_docs}.
  \end{itemize}

  \paragraph{Synchronous vs asynchronous UX.}
  For small computations, the wrapper can call the service synchronously.
  For heavy tasks, the wrapper returns immediately with a \texttt{job\_id}
  and a status endpoint; results are stored as S3 artifacts and referenced
  by metadata. This avoids API Gateway timeout constraints while
  preserving a clean user experience.

  \paragraph{Why.}
  Optima is a proof that the control plane is not tied to a single execution
  substrate. The same governance primitives—module gating, effective context,
  scope envelope, auditability—apply whether execution happens in a Lambda
  query plane (Horizon/Grid), a Kubernetes service (DataMap), a connector
  worker fleet, or a heavy compute service (Optima). The platform is cohesive
  because governance is cohesive.
