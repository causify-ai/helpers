  \section{Problem Statement and Design Goals}
  \label{sec:problem}
% ==============================================================================
  \subsection{The problem: governance smeared across the runtime}
  In multi-tenant systems, ``tenant'' is both a business concept and a security
  boundary. The main failure mode is not a spectacular vulnerability; it
  is \emph{inconsistency}: one component thinks the tenant is \texttt{acme},
  another component thinks it is \texttt{demo-acme}, and a third component
  accepts a user-provided \texttt{schema} hint.

  This creates three recurring classes of incidents:
  \begin{itemize}
    \item \textbf{Policy sprawl:} authorization and capability checks
      scattered across services, gateways, and UI layers.

    \item \textbf{Confused deputy:} a service acts on behalf of a tenant
      it did not intend to, because it trusted caller-provided scoping parameters
      \cite{confused_deputy}.

    \item \textbf{Surface drift:} documentation and client behavior
      diverge from what is actually enabled/allowed (e.g., stale OpenAPI
      specs or endpoints that exist but should not be reachable).
  \end{itemize}

  Kubernetes and container orchestrators are excellent at running
  workloads. They are not optimized to be the single source of truth for
  tenant identity and policy. When the cluster becomes the governance
  layer, every runtime change becomes a policy event. That is how you
  get ``just one more admission policy'' until nothing moves.

  \subsection{Design goals}
  We adopt five goals that reflect both engineering and operations:

  \begin{enumerate}
    \item \textbf{G1 --- Explicit context:} every request has an
      immutable \emph{actor} identity and an explicit \emph{effective
      context}. If \viewas{} exists, it is visible and auditable.

    \item \textbf{G2 --- Server-injected scoping:} tenant scoping
      signals (e.g., schema prefixes) are derived by the control plane
      and injected downstream. Client-supplied scoping parameters are treated
      as untrusted input.

    \item \textbf{G3 --- Minimized surface:} the platform should not
      advertise APIs and modules that are not enabled for the effective
      context. OpenAPI and endpoint access are filtered accordingly.

    \item \textbf{G4 --- Cache-friendly contracts:} context contracts
      such as \texttt{/v2/me} should support conditional requests (ETag/\texttt{If-None-Match})
      to reduce repeated payload transfer \cite{rfc9110}.

    \item \textbf{G5 --- Operational clarity:} logs and traces must
      encode both actor and effective tenant context so that incidents
      can be debugged without guessing.
  \end{enumerate}

  These goals are intentionally conservative. Multi-tenant platforms do
  not need cleverness. They need consistency. Cleverness is reserved for
  the causal models.
