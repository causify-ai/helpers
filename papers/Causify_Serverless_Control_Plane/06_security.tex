  \section{Security and Defense-in-Depth}
  \label{sec:security} A multi-tenant platform's primary security objective
  is not "no bugs." It is: \emph{no tenant confusion}. Most catastrophic
  failures start as mundane mis-scoping.

  We therefore design defenses around a concrete threat model.

% ==============================================================================
  \subsection{Threat model (what we actually defend against)}
  We focus on:
  \begin{itemize}
    \item \textbf{Forged tenancy context:} a client attempts to select a
      tenant/role they do not own.

    \item \textbf{Confused deputy:} a downstream service performs
      actions for the wrong tenant because it trusted caller-provided
      scope.

    \item \textbf{Surface drift:} clients discover and call endpoints
      that are not intended to exist for their effective context.

    \item \textbf{Accidental operator damage:} support/QA performs real
      actions while impersonating the wrong tenant.
  \end{itemize}

% ==============================================================================
  \subsection{Audit-Ready Foundations}
  This paper is not a compliance report, but production reality is
  constrained by compliance-grade expectations: clear boundaries, clear
  evidence, and recoverability that works under pressure. Those
  constraints are not paperwork; they are architecture.

  \paragraph{Blast-radius discipline (accounts and regions).}
  We operate with explicit separation boundaries (e.g., production vs.\ non-production
  isolation and regional separation for resilience). The key point is
  not the exact account topology; it is the design intent: reduce blast
  radius, simplify incident containment, and make boundaries legible to
  both operators and auditors.

  \paragraph{Network segmentation as a default posture.}
  Ingress is terminated at controlled edges (load balancers and WAF
  where applicable), while compute and data stores remain in private
  subnets without public IPs. When a platform is multi-tenant, ``private
  by default'' is not a preference; it is a survival trait.

  \paragraph{Recoverability is engineered, not enabled.}
  Backups exist at multiple layers (databases, object storage, volumes,
  and file systems), but the important detail is restore: we treat restore
  as a first-class operational capability and validate it regularly. In
  multi-tenant platforms, recoverability is part of tenant isolation:
  mistakes must be reversible.

  \paragraph{Evidence is just observability with manners.}
  Operational telemetry (logs, metrics, traces, configuration history)
  doubles as evidence when it is structured, retained, and attributable.
  That attribution is exactly why this paper emphasizes actor vs.\ effective
  context: it makes the system auditable by design, not by spreadsheet.

  \subsection{Control plane hardening: explicitness over cleverness}
  The control plane adds baseline security headers on every response:
  \texttt{X-Frame-Options: DENY}, strict CSP (\texttt{default-src 'none'}),
  \texttt{X-Content-Type-Options: nosniff}, and HSTS. These are not glamorous,
  but they remove entire classes of browser-side surprises.

  It also enforces that \viewas{} must contain \texttt{tenant\_id} and
  \texttt{role}, and denies requests that attempt to impersonate targets
  outside the allow-list. This turns impersonation from "header magic" into
  a controlled capability.

% ==============================================================================
  \subsection{Scope injection: the anti-confused-deputy pattern}
  The most important security property of this platform is not that it denies
  access. It is that it \emph{denies ambiguity}. When the control plane
  injects \texttt{schema\_prefix} into downstream requests, it removes a
  class of vulnerabilities where the client "helpfully" tells the system
  what tenant it meant.

  Downstream services can still validate scope, but they no longer need to
  guess it.

% ==============================================================================
  \subsection{Kubernetes edge security: internal ingress + WAF + audit logs}
  Kubernetes ingress definitions for services like the Console and Airflow
  webserver use internal ALBs, enforce TLS policies, attach a regional WAF
  ACL, enable Shield, and stream access/connection logs to S3. This
  provides:
  \begin{itemize}
    \item reduced exposure (internal-only),

    \item consistent layer-7 protection (WAF),

    \item DDoS hardening (Shield),

    \item and an audit trail for forensic analysis (ALB logs).
  \end{itemize}

% ==============================================================================
  \subsection{Secrets and workload identity: IRSA/ESO for Kubernetes}
  Workloads obtain secrets via ESO and IRSA: service accounts are annotated
  with an IAM role ARN, enabling controlled access to Secrets Manager
  without embedding long-lived credentials in pods. This keeps secret access
  auditable and reduces credential sprawl.

% ==============================================================================
  \subsection{Caveat: caching and multi-tenant correctness}
  ETags and short TTL caching make context contracts fast and cheap \cite{rfc9110}.
  However, in multi-tenant systems, caching must never blur identity
  boundaries. We therefore treat these contracts as \emph{safe for
  revalidation} and ensure the ETag is derived from the full response
  body (which includes effective context). In environments with shared intermediaries,
  additional cache controls (e.g., \texttt{private}) may be appropriate;
  the key invariant is: cached responses must not be reusable across principals.

% ==============================================================================
  \subsection{Summary: security is mostly architecture}
  The security story here is not a single trick. It is a refusal to
  allow tenancy to be implicit. The control plane makes context explicit,
  the data plane executes within that context, and the edges are hardened
  so mistakes are contained.

  Or, in operational terms: fewer "how did this happen?" incidents, and more
  "this couldn't have happened" guarantees.
