\section{System Overview}
\label{sec:system_overview}

\uiPolicy{} has one moving part that matters more than the rest: the backend materializes a UI contract before the web tier renders the product. The contract is scoped to the authenticated actor, the effective tenant, and the effective role. The browser receives a product surface that has already been shaped by policy.

The key response is step 6. \code{/v2/me} is not just a profile endpoint. It is the control-plane contract that tells the web tier which tenant world it should render.

\subsection{The UI contract}
\label{sec:ui_contract}

A simplified response looks like this.

\Needspace{22\baselineskip}
\begin{lstlisting}[style=json, caption={Representative \code{/v2/me} response for a two-module tenant.}, label={lst:v2me}]
{
  "actor": {
    "sub": "u123",
    "email": "engineer@example.com"
  },
  "view_as": {
    "tenant_id": "acme",
    "role": "admin",
    "modules": ["horizon", "grid"],
    "module_layout": { "sidebar": "full" }
  },
  "impersonatable": [
    {"tenant_id": "acme", "roles": ["member", "admin"]},
    {"tenant_id": "globex", "roles": ["member"]}
  ]
}
\end{lstlisting}

The fields separate two identities that are often blurred.

\subsection{Module structure}
\label{sec:modules}

The platform defines seven independently evolvable modules: Horizon, Grid, Optima, Panorama, Frontier, Sentinel, and DataMap. Each module follows the same shape.

\begin{itemize}
  \item A server layout checks the effective ability before rendering.
  \item A client module renders the experience selected by the contract.
  \item API routes inherit the same effective context from the server session.
  \item Module-specific layout is fetched only after a server-side ability check.
\end{itemize}

That last point is important. Layout is treated as privileged data. If a user cannot view a module, the server will not return its layout even if a modified browser requests it directly.
