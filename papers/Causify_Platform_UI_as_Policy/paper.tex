  \date{\today}

  \begin{abstract}
    Multi-tenant SaaS products tend to develop a particular kind of split-brain:
    the backend enforces permissions, while the frontend \emph{guesses}
    what the user should be able to see. Over time, those guesses drift.
    The UI becomes a polite liar (buttons that 403) and occasionally a dangerous
    liar (surfaces that imply access that should never exist).

    We present \uiPolicy{}: a capability-driven architecture where the UI
    is treated as a first-class policy surface. The central idea is
    \uiData{}: the backend materializes a tenant- and role-scoped \emph{UI
    contract} (enabled modules and structured layout fragments) and
    delivers it at runtime. The web tier then composes navigation and module
    experiences from this contract while deriving an identical authorization
    model server-side and exporting it to the client in a packed form for
    UX consistency.

    Our production implementation couples (i) a control plane API (AWS
    Lambda) that returns \texttt{/v2/me} with an explicit \emph{actor} vs.\ \emph{\viewas}
    split, (ii) tenant policy stored as data in DynamoDB (modules and layout
    fragments), and (iii) server-authoritative abilities (CASL) serialized
    for the client. This design reduces tenant onboarding friction (configuration
    over redeploy), minimizes frontend/backend authorization drift, and
    enables safe support impersonation without sacrificing auditability.
  \end{abstract}
